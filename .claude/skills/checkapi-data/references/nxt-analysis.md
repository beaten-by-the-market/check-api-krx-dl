# NXT(넥스트레이드) 분석 — 거래소별 패밀리·유니버스·상폐 종목

2026-07-09~10 실호출로 전수 검증. 헬퍼: `scripts/nxt_universe.py`.

## 거래소별 stock 패밀리 (실측 확정)

| 패밀리 | 시장 | 비고 |
| --- | --- | --- |
| m001 / m003 | KRX 코스피 / 코스닥 | |
| **m222 / m223** | **NXT 코스피 / 코스닥** | NXT **단독 체결분** |
| **m224 / m225** | **통합 코스피 / 코스닥** | KRX + NXT |
| m226~m229 | NXT/통합 **업종** | 종목 아님(업종 지수) |

**live 데이터가 실제로 나온다** (과거 "spec state=off라 반환 여부 불명" 서술 정정).
005930, 2026-07-08 실측 — 거래량이 정확히 합산되고 종가는 거래소별로 다르다:

```
m001(KRX) 33,525,758 + m222(NXT) 33,716,349 = m224(통합) 67,242,107
종가:      277,500          267,000                     267,000
```

→ **거래소별 분석은 m222/m223, 시장 전체는 m224/m225.** m001만 보면 NXT 물량이 통째로 빠진다.

## NXT 유니버스 (실측)

- **첫 거래일 = 2025-03-24.** 출범일(2025-03-04)엔 각 시장 **5종목씩만** 시범 거래.
  `sdate=20250301`로 불러도 그 이전 행은 안 나온다.
- **NXT 등록 ≠ NXT 거래.** code_info는 코스피 835 / 코스닥 1,817이지만, 하루 실제 체결은
  코스피 ~336 / 코스닥 ~269 (2026-07 기준). 나머지는 거래량이 **진짜 0**이다.
- 전 기간(2025-03-24~2026-07-09) 한 번이라도 거래된 고유종목 = **1,009개**(코스피 459 + 코스닥 550).
- **거래 종목 집합은 매우 안정적**이다. 330거래일 중 218일이 직전일과 집합이 **완전히 동일**하다.
  (버그로 오해하기 쉬움 — 표본 검증에서 거짓양성·누락 0으로 확인됨.)

**ETP는 NXT에 없다.** 증권그룹ID(F34501) 분포 — ETF(EF)·ETN(EN)이 아예 없어 별도 필터가 불필요:

| 시장 | 구성 |
| --- | --- |
| m222 | ST 807 · RT 23(리츠) · IF 2 · MF 1 · FS 1 · DR 1 |
| m223 | ST 1,798 · FS 11(외국주식) · DR 8 |

보통주만 원하면 리츠·인프라펀드·외국주식/DR을 `F34501`로 걸러낸다.

## 상장폐지 종목을 포함한 과거 유니버스 (핵심 함정)

**`code_info`는 현재 상장분만 준다.** 코드 카탈로그(35,340개, `search_codes.py`)에도 상폐분은 없다.
그래서 code_info로 과거 유니버스를 만들면 **당시 거래되던 상폐 종목이 통째로 누락**된다.

해법은 두 가지이고, 둘 다 실측 확인됐다:

1. **`rank_invest_date` 는 `sdate`/`edate` 시점의 유니버스를 준다.**
   신성통상(005390, 2025-09-29 상폐): 2025-09-01 조회엔 포함, 2025-10-01 조회엔 없음.
2. **`hist_info` 는 상폐 종목의 과거 시세를 그대로 보관한다.**
   005390 → m222에서 125행(2025-03-31~09-29) 반환. 2024-07 상폐된 003410도 m001에서 1,114행 반환.

→ 과거 유니버스가 필요하면 `rank_invest_date`를 **월 단위로 훑어 합집합**을 만든다.

**"상장 ≠ 거래" 를 다시 구분할 것.** rank_invest_date의 행 자체는 그날 **상장돼 있던** 종목이다.
2025-03-04에 848행이 오지만 실제 체결은 5종목뿐이다. 실거래 판별은 아래 두 방법 중 하나:

- `rank_invest_date` 의 **매도거래량 `F06505_12` 또는 매수거래량 `F06507_12` > 0** (전체=투자자번호 12)
- `hist_info` 의 **거래량 `F15015` > 0**

두 방법은 실측에서 **완전히 일치**한다(2026-07-02·03, 각 50종목 층화표본, 거짓양성 0 / 누락 0).

⚠ **매수거래량은 `F06507`이다. `F06506`은 존재하지 않는다** (매도 F06505 / 매수 F06507 / 순매수 F06508).

## 전종목×날짜 endpoint는 다섯 개뿐 — 그리고 종가는 없다

/stock 747개 명세 전수 스캔 결과, **종목코드 없이 날짜만 받는** endpoint:

| endpoint | 필드 | 내용 | 종가 |
| --- | --- | --- | --- |
| `rank_short_date` | 7 | 공매도 기간합산 | ✗ |
| `member_date` | 10 | 회원사 매매집계 | ✗ |
| `program_all` | 22 | 프로그램매매 (NXT는 0행) | ✗ |
| `rank_mass_date` | 33 | 대량매매 (m001/m003만) | ✗ |
| `rank_invest_date` | **123** | 투자자별 매매 | ✗ |

**과거 일별 전종목 시세를 한 번에 주는 endpoint는 없다.** `rank`(15필드)가 전종목 시세를 주지만
**날짜 파라미터가 없어 당일만** 된다. 과거 시세는 종목별 `hist_info` 반복이 유일한 길이다.

## 일별 NXT 실거래 유니버스 뽑기 (권장 레시피)

```bash
python .claude/skills/checkapi-data/scripts/nxt_universe.py --probe   # data_list 지원 확인
python .claude/skills/checkapi-data/scripts/nxt_universe.py           # 수집 (등록 IP·샌드박스 밖)
```

거래일마다 `rank_invest_date(sdate=edate=D, data_list=F16013,F06505_12,F06507_12)`를 부르고
거래량 > 0 만 남긴다. 662회 호출 / **48MB** / 약 3분에 2025-03-04~2026-07-09 전 기간이 나온다.

`data_list` 없이 같은 걸 돌리면 **1.1GB로 일 사용량 한도(1e9 bytes)를 초과**한다. 실제로 그렇게
날려먹은 적이 있다. 아래 "용량·한도" 절을 반드시 볼 것.

산출: `(일자, 시장, 단축코드, 종목명, 증권그룹, 현재상장여부)` + 일자별 거래종목수 요약.
상폐 종목은 code_info에 없어 **종목명이 빈칸**이다(코드는 정확). 이름이 필요하면
`/etc/comp/comp_basic`(`DELIST_DT` 보유)으로 보완한다.

## 용량·한도·rate limit (뒤늦게 발견한 제약들)

- **일 사용량 한도 = 1,000,000,000 bytes(=1GB, `cust_id` 단위).** 초과 시 모든 endpoint가
  `일 최대 사용량 1,000,000,000Bytes를 초과했습니다.` 로 실패한다. **호출 수가 아니라 응답 바이트가
  아껴야 할 자원이다.** (662회 호출 = 1.1GB / 2,652회 호출 = 175MB 인 역전이 실제로 일어난다.)
- **`data_list` 로 필드를 골라 받으면 용량이 수십 분의 1이 된다.** 123필드 → 3필드에서 1.1GB → 48MB.
- **`data_list` 는 NXT 명세에 미기재지만 실제로 동작한다.** m001엔 22/35 endpoint에 문서화돼 있는데
  m222엔 2/23뿐 — 명세 누락이다. `rank_invest_date`·`hist_info` 모두 NXT에서 정상 동작 확인.
- ⚠ **`data_list` 는 존재하지 않는 F-code를 오류 없이 조용히 버린다.** `F06506_12`(없는 코드)를
  넣었더니 경고 없이 2개 필드만 왔다. **요청한 개수와 반환된 개수를 반드시 대조**할 것.
- **시계열 조회는 초당 1회 rate limit.** `시계열 데이터의 조회는 1초에 1회로 제한됩니다.`
  `hist_info` 등을 빠르게 반복하면 걸린다 → 호출 간 **≥1.15초 간격**.
- **`rank_invest_date` 조회기간은 1년 이내.** 초과 시 `param_denied` / `조회기간은 1년을 초과할 수 없습니다.`
- **범위가 크면 HTTP 502.** 1년치 rank_invest_date 단일 호출은 502. 월 단위로 쪼갤 것.

## 한도 초과 vs IP 차단 구분

메시지가 다르므로 헷갈리지 말 것:

| 상황 | message |
| --- | --- |
| 일 사용량 초과 | `일 최대 사용량 1,000,000,000Bytes를 초과했습니다.` |
| 키 오류 **또는 미등록 IP** | `cust_id 또는 auth_key가 정확하지 않습니다.` |

전자가 오면 인증·IP는 정상이고 계량기만 막힌 것이다. 한도는 일 단위로 리셋된다(실측).

## `success:false` 를 빈 결과로 처리하지 말 것

한도 초과·rate limit은 HTTP 200 + `success:false` 로 온다. 이걸 `results or []` 로 흘리면
**"그날 거래종목 0개"로 조용히 기록**된다. 실제로 이 버그 때문에 코스닥 305일이 빈 값으로 채워진 채
"완료"가 찍혔다. 반드시 예외로 올리고, 완료 시 **시장별 커버 거래일 수를 출력해 완전성을 검사**한다.
